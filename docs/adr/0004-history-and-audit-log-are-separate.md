# ADR 0004: History and Audit Log are separate persistence mechanisms

## Status

Accepted

## Context

SQLcery needs to persist executed Statements for two distinct purposes:

- **Interactive recall** — the user presses `Ctrl-r` and fuzzy-searches recent Statements to restore one into the Command Pane. This is connection-scoped: when working against a given database, you want to recall what you ran against *that* database, not every Statement ever executed across all connections.
- **Audit trail** — a complete, ordered record of every Statement executed, across all Connections, for compliance, debugging, or external tooling. This is global: connection boundary is irrelevant.

These two use cases have incompatible access patterns. A single unified log optimized for audit (flat, append-only, cross-connection) is the wrong structure for interactive recall — the user would need to filter by connection on every `Ctrl-r` invocation. A per-connection history optimized for recall is the wrong structure for audit — there is no single file to tail, grep, or ingest.

## Decision

Maintain two separate persistence mechanisms:

- **History** — per-Connection (or per-Connection String). Persists submitted Statements across Sessions to the same Connection and is used exclusively for interactive recall via `Ctrl-r`. Successful, failed, cancelled, and timed-out Statements are all retained so incomplete attempts can be edited and retried. Repeated submission of identical Statement text produces one History entry: the latest occurrence replaces the earlier occurrence and moves to the most-recent position. Deduplication compares exact text, so whitespace-different Statements remain distinct. Each Connection identity retains at most its 1,000 most recent unique Statements. Slash Commands are never persisted to either store because they never execute against the database — they only open application UI or expand a Statement into the Command Pane; the expanded Statement is recorded if and when the user submits it. (This deliberately drops the prior behaviour of logging failed or non-expanding Slash Commands to the combined log. The exclusion is anchored to the "never executes against the database" invariant, not to the word "Statement": a future Slash Command that runs SQL directly would have to emit audit events.) A named Connection is identified by its configuration origin plus its name: global and project-local definitions do not share History, and same-named definitions in different projects remain separate even when they currently resolve to the same database. Project origins use canonical absolute paths. A direct Connection String is identified by a SHA-256 hash of its exact supplied text: the Connection String is never exposed in the storage path, and textually different Connection Strings do not share History. History is stored as an ordered JSON array of Statement strings in `$XDG_DATA_HOME/sqlcery/history/<connection_id>.json`; it contains no Audit Log metadata. The complete file is replaced atomically after each change so deduplication and the size cap also bound physical storage. Storage filenames are hashes of Connection identities rather than user-supplied names or paths.
- **Audit Log** — a single flat append-only JSON file at `$XDG_DATA_HOME/sqlcery/audit.log`. Each Statement execution is represented by two events correlated by an execution identity. A `submitted` event is appended and durably synced before database execution; it contains the connection name, connection origin, opaque connection identity, Statement text, and submission timestamp. The `submitted` event is written only once SQLcery commits a complete Statement to a live Session, immediately before handing it to the database. A Statement that is never attempted against a live Session — because there is no Session, or because client-side completeness gating rejected it — produces no audit events, since there is no execution and no connection identity to attribute it to. A `completed` event is appended and durably synced after execution; it contains the terminal timestamp, outcome, and result summary. The terminal outcome is one of `success`, `failure`, `cancelled`, or `timed_out`. Result summaries are metadata only: query row count, affected-row count, generic success status, or error text (bounded as described below). Result rows and cell values are never logged. The terminal outcome and metadata summary (row counts, affected-row count, generic success status) are always recorded in full; only the human-readable error string is bounded by a sanity cap (a few KB, explicitly marked when truncated) so a pathological driver error cannot bloat the unrotated log or threaten the fail-closed write path. Knowing that a Statement failed is never lost — only the verbose error string may be shortened. A `submitted` event without a matching `completed` event means the outcome is unknown because SQLcery was interrupted; no terminal outcome is inferred. For a project-local named Connection, the origin is the canonical absolute path to its `connections.toml`; exposing that path in the Audit Log is intentional so external tooling can distinguish same-named Connections from different projects. A global named Connection records its global configuration origin. A direct Connection String records `direct` as its origin and its hashed identity, never the raw Connection String. Retention is unbounded: SQLcery does not rotate, truncate, or delete audit events.

## Consequences

- `Ctrl-r` search is fast and relevant — no cross-connection noise.
- Implementing either store requires threading connection *origin* through config resolution as groundwork. Today both the global and project `connections.toml` are decoded into a single merged map, discarding which file each named Connection came from; the loader must instead record per-Connection provenance. A single shared `ConnectionIdentity` — origin + name for a named Connection, or a SHA-256 of the raw text for a direct Connection String — is derived once and consumed by *both* stores, so their scoping cannot drift.
- The durable audit writes (`submitted` before execution, `completed` after) run on the execution worker goroutine owned by the execution coordinator, never on the UI loop. The UI reacts only to messages, so an fsync can never freeze the TUI.
- Recovery from a blocked `completed` write is enforced at the next submit: the following Enter first attempts to flush the pending event and proceeds only if it succeeds. Because execution is blocked while one event is pending, there is at most one unwritten `completed` event — a single slot, not a queue. Quitting is never blocked; it abandons the in-memory event, intentionally leaving an unmatched `submitted` whose outcome is unknown.
- Interactive recall is served from an in-memory snapshot loaded at Session startup plus the current Session's own additions; it does not live-merge Statements written by other concurrent Sessions until restart. The on-disk merge-on-write still guarantees no Statement is lost durably.
- History suppresses exact duplicates for recall; the Audit Log retains both events for every submission without deduplication.
- History storage is bounded to 1,000 unique Statements per Connection identity; Audit Log retention remains unbounded.
- Atomic replacement keeps each bounded History file internally consistent; Audit Log remains independently append-only.
- Audit persistence is fail-closed. SQLcery does not execute a Statement until its `submitted` event is durably written. If the `completed` event cannot be durably written after execution, SQLcery retains that event in memory, presents a persistent error, and blocks subsequent Statement execution until it is written. History persistence remains independent and must still be attempted.
- Audit persistence is mandatory and has no configuration switch. Disabling it would make the completeness and fail-closed guarantees conditional.
- History persistence is fail-open. If its atomic replacement fails, SQLcery reports the error but permits subsequent Statement execution; the current Session's in-memory History still receives the Statement and remains available for recall. A later successful rewrite persists the current bounded in-memory History.
- Persistence is safe across concurrent SQLcery processes sharing the same data directory. Each Audit Log append is serialized with an inter-process lock; events from different executions may interleave and are correlated by execution identity. Each History update locks its per-identity store, reloads the latest file, merges the new Statement, deduplicates and caps the entries, and then atomically replaces the file so concurrent sessions do not lose one another's updates. The lock is an advisory lock taken on the target file's own descriptor (released automatically by the OS on process death, so a crash cannot leave a stale lock) via a small cross-platform flock library, giving correct behaviour on Windows as well as Unix.
- No migration compatibility is provided for the existing development-format persistence. Before using the new mechanisms, delete `audit.log`, `audit.log.1`, and any persistence produced by the old combined History/Audit implementation. The new Audit Log and each new per-identity History start empty. This destructive reset is acceptable only because the format is being replaced before a stable release; no legacy parser or import path is implemented.
- The Audit Log is a simple file that external tools (grep, jq, log shippers) can consume without understanding SQLcery internals.
- Audit Log storage grows without an application-managed limit. Retention and archival are the user's or external tooling's responsibility.
- Two storage paths to maintain. They are not in sync by design — deleting or resetting History storage does not affect the Audit Log. A user-facing clear-history command is outside this ADR.
- The Audit Log and History are permanently separate; neither is intended to absorb the other.
